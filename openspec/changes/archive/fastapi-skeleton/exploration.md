## Approaches

### 1. Minimal core install (RECOMMENDED for this slice)

Pin only `fastapi~=0.116` + `httpx~=0.27` in `dependencies`. Skip `uvicorn`, `jinja2`, `python-multipart`, `email-validator`, `pydantic-settings`, `pydantic-extra-types`, `fastapi-cli` for now. Use `from fastapi.testclient import TestClient` for the test.

- **Pros**:
  - Smallest possible dep tree for the smallest possible slice. Reviewer-friendly diff.
  - 80% coverage trivially satisfied: one route, one test, both 100% exercised.
  - No new pattern to establish yet (no lifespan, no middleware, no CORS).
  - mypy --strict is well-supported by FastAPI's Annotated-style signatures.
  - httpx is the only transitive runtime risk; ~0.27 is the broadest well-tested range.
- **Cons**:
  - Cannot actually `uvicorn app.main:app` locally without adding uvicorn later. Slicing fine, but the "Run it" story is incomplete.
  - httpx 0.27 emits a deprecation warning about the `app` shortcut (cosmetic, not a failure); looks noisy in pytest output.
- **Effort**: Low. ~30 lines of code + ~50 lines of lockfile churn.

### 2. `fastapi[standard]` install (defer to a later slice)

Install `fastapi[standard]` which pulls in `fastapi-cli`, `fastapi-cloud-cli`, `uvicorn[standard]`, `httpx`, `jinja2`, `python-multipart`, `email-validator`, `pydantic-settings`, `pydantic-extra-types`.

- **Pros**:
  - "Batteries-included" — `fastapi dev`, `fastapi deploy`, uvicorn, form parsing, email validation all work out of the box.
  - Aligns with FastAPI's official documentation (the `pip install "fastapi[standard]"` recommendation).
  - Lets us add `make run` in the same slice if we want.
- **Cons**:
  - Pulls in 20+ transitive packages (uvicorn, uvloop, watchfiles, rich, click, shellingham, typer, pydantic-settings, email-validator, jinja2, python-multipart, itsdangerous, pyyaml, …). Bigger dep tree, bigger `uv.lock` diff, slower lockfile review.
  - `fastapi-cloud-cli` and `fastapi-cli` are tooling, not runtime; increases the dep-tree surface for a slice that doesn't use them.
  - More things to vet on mypy --strict (e.g., `uvloop` on Windows — not actually loaded by the test, but in the dep graph).
  - The `cloud-cli` dep is vendor-specific (FastAPI Cloud) and feels premature for a solo-dev Windows-trading project.
- **Effort**: Medium. Same code, but the dep review and lockfile churn is ~5x larger.

### 3. Core + uvicorn (middle ground)

Install `fastapi~=0.116` + `httpx~=0.27` + `uvicorn[standard]~=0.32` so `uv run uvicorn app.main:app --reload` works.

- **Pros**:
  - "Run it" story is complete in the same slice.
  - Smaller tree than `[standard]` (skips cli/jinja/email/pydantic-settings).
  - Reviewer can actually exercise the endpoint locally.
- **Cons**:
  - Still premature for a strict TDD slice that only requires a test to pass.
  - Slightly larger lockfile churn than Option 1.
  - "Out of scope" risk: the user's confirmed scope for this slice is FastAPI + /health + TestClient. uvicorn for "run" is a separate concern that can land when there's an actual run target (Makefile `run` target, not this slice).
- **Effort**: Low-Medium. ~+10 lockfile entries vs Option 1.

## Recommendation

**Option 1 (minimal core install)** for this slice. Reasons:

1. **Strict scope match.** The user-confirmed scope is "FastAPI + /health + TestClient test + fastapi in pyproject.toml". uvicorn, jinja, email, pydantic-settings, fastapi-cli are all out of scope for a skeleton slice. Adding them inflates review surface for zero functional gain.
2. **Smallest possible diff for the first real code in the repo.** 30 lines of code + a lockfile bump. Reviewer can read it in one pass.
3. **Strict TDD binds cleanly.** One test, one route, one assert. RED → GREEN → REFACTOR is one cycle.
4. **Coverage gate becomes a real signal for the first time.** The 100%-of-0 problem (vacuous coverage) goes away: 1 route, 1 test, real %.
5. **Dependabot will roll forward.** Once `fastapi` and `httpx` are in `pyproject.toml` with `~=` pins, the existing `uv` ecosystem entry in `.github/dependabot.yml` will propose patch bumps. Future slices can swap to `fastapi[standard]` once we actually need uvicorn / cli / forms / settings.
6. **httpx 0.27 deprecation warning is acceptable.** It's a `DeprecationWarning` on the test client instantiation, not a test failure. Will be a SUGGESTION in verify but not a blocker. (Documented in AGENTS.md as a known cosmetic if it becomes annoying.)

**Version pin choices** (proposal phase can confirm):

- `fastapi~=0.116` — known-stable line (released 2025-07-11 for 0.116.1, survived 2+ months of subsequent releases). Avoids the 0.13x beta channel. Allows patches (0.116.x) but blocks breaking 0.117+.
- `httpx~=0.27` — broadest range that still triggers the deprecation warning we can document. Allows 0.27.x patches. Avoids the very latest 0.28.x line which has different transitive behavior.

**Alternative stable pins** (if the proposal wants a newer line):

- `fastapi~=0.118` (Oct 2025) or `fastapi~=0.124` (Dec 2025) — newer, more features, still <1.0.
- `httpx~=0.28` — latest 0.x, no deprecation warning, but tighter range means more Dependabot churn.

## Risks

- **httpx 0.27 deprecation warning on TestClient.** Cosmetic `DeprecationWarning: The 'app' shortcut is now deprecated. Use the explicit style 'transport=WSGITransport(app=...)' instead.` Emitted at test client construction. Not a test failure. Will appear in `pytest -W error` mode if someone turns on strict warnings; default pytest does not. **Mitigation**: pin `httpx~=0.28` if a clean test output matters; or accept the warning and document in AGENTS.md.
- **FastAPI beta channel drift.** The 0.13x line is marked beta on PyPI. Pinning to `~=0.116` (a known-stable minor) sidesteps this. If the proposal picks a different minor, double-check the release notes for regressions.
- **mypy --strict on FastAPI annotations.** FastAPI 0.116+ uses `Annotated[...]` for path/query/body params, which is mypy-friendly. The bare `app = FastAPI()` annotation is `app: FastAPI`. Both should pass strict. **Mitigation**: if a mypy surprise appears, add the `pydantic.mypy` plugin to `pyproject.toml` `[tool.mypy]` — well-supported, mature.
- **Lockfile drift in the same commit.** `uv add` regenerates `uv.lock`. The apply phase MUST commit `uv.lock` in the **same work unit** as the `pyproject.toml` dep change (precedent: dev-tooling apply did this successfully). `uv lock --check` is a binding CI step.
- **Pydantic version drift.** FastAPI's pydantic pin is `>=2.9.0`; not directly pinned in our `pyproject.toml`. The transitive version is locked in `uv.lock`. No action needed; will be tracked by Dependabot.
- **Coverage drop risk if app code is added without tests.** Low for this slice (1 route, 1 test, 100%). Becomes real once more routes/models are added. **Mitigation**: enforce the 80% gate per `--cov-fail-under=80` in `[tool.pytest.ini_options]` (already in place). Coverage is not vacuous anymore.
- **Dependabot opens PRs immediately after merge.** Once `fastapi` and `httpx` are in `pyproject.toml`, the next weekly Dependabot run will scan them. With `~=` pins the bumps are patch-only. Will auto-merge on `Quality Gate` green (per the existing `dev-tooling` setup).
- **Ruff S (security) rule on new code.** FastAPI's `FastAPI()` constructor and `TestClient(app)` calls should not trigger `S` rules. The `tests/** = S101` per-file ignore is already in place for pytest asserts. No additional overrides needed.
- **Stale `cpython-313.pyc` in `app/__pycache__/`.** Leftover from a previous Python 3.13 run; the active venv is 3.12. Not relevant to this slice. (Future cleanup, if it ever shows up in `git status`, would be a one-line `git clean -fd app/__pycache__`.)
- **Windows PowerShell `RemoteException` on `uv` commands.** Carried over from `AGENTS.md` and the prior 3 cycles. Still a terminal artifact, not a real failure. Trust the exit code.
- **`make` not installed on Windows PowerShell.** If the apply phase needs to run `make ci`, it must use Git Bash/WSL or the raw `uv run` commands. The Makefile is unchanged; this slice doesn't add a `make run` target.
